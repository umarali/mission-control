0005 — v2 Slack collector (buried items)

2026-06-23 · Pure core shipped; live wiring handed off · Umar + Claude

area: api + web PLAN.md §8, pain #3 issues: #16 (slice b)

Problem

v2's second slice (#16): the first collaboration surface. PLAN.md's pain #3 is "important Slack / Asana / GitHub items get buried." Slack is first. The catch (flagged in the 0003 handoff): the app can only read Slack with the owner's runtime token, which the ephemeral cloud container doesn't have — so the live fetch can't be verified here. The owner chose to build the part that is verifiable now: the pure parser + event derivation + the Timeline rendering, grounded in real-shaped fixtures.

Solution

api/missioncontrol/slack.py — pure, credential-free, same shape as transcript.py/codex.py. Input is an already-parsed read-only Slack Web API response; the module never calls Slack and never touches a token.

  • What's a "buried item". A DM, or a message that @-mentions you. Your own messages, system/join notices, and bot posts are explicitly not buried items and are skipped. parse_history handles conversations.history (DM channel → every message from someone else is a DM; regular channel → only <@self> mentions). parse_search handles search.messages (mentions, with channel + permalink).
  • Readable, safe text. Each message is demangled from Slack mrkdwn (<@U123|al>→@al, <#C1|eng>→ #eng, <url|label>→label) and then run through redact_text — a pasted token can never reach the store or the browser.
  • Tolerant (C5). Non-dict/auth-failed/partial responses return a degraded feed (with the redacted Slack error), never a crash. An available but empty feed ("Slack reachable, nothing buried") is distinct from a degraded one.
  • Into the Timeline. slack_events derives scrubbed Event rows (surface slack, type slack.dm/ slack.mention) that entry 0004's Timeline already renders generically; a slack filter pill appears automatically (vendor-neutral). The web summary added a slack.* branch → "DM from Alice: …" / "bob in #eng: …". slack_ts (the original message ts) is carried as the per-message idempotency key so the eventual live loop can dedupe across ticks.
Verified now: real-shaped, sanitized fixtures (DM, channel mention, search match) incl. degraded/auth/redaction/demangle cases — +8 tests (103 total), slack.py 100% line+branch; ruff + mypy + web lint/typecheck clean. Live via Playwright: a DM and a mention render correctly in the Timeline with demangled, redacted text.

Context

What's left (handed to the owner, needs the runtime token): a thin token-gated reader seam — read SLACK_TOKEN at call time (in memory only, redacted from logs), httpx read-only GETs to conversations.list(types=im) + conversations.history and/or search.messages (search needs a user token), a short timeout + backoff — then feed the raw JSON to parse_history/parse_search and append slack_events(feed) to the store in record_snapshot, deduping on slack_ts. It belongs in the coverage-omit seam list (like paths.py/osnotify.py) and should be opt-in (e.g. only attempt when SLACK_TOKEN is set) so the non-invasive default holds. The fixtures in test_slack.py are exactly the shapes that reader must produce, so wiring is mechanical + verifiable once a token exists.

Then (still v2/v3): the generalized guarded action endpoint — best built with Slack's first "act" action (reply / mark-read) so the abstraction has >1 user (the /api/jump token+Origin pattern is the template). Then v3 (#17) work-threads and AWS (#18).
