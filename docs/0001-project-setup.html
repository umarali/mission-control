0001 — Project scoping & setup

2026-06-21 to 2026-06-23 · Done (build not started) · Umar + Claude

area: planning + repo setup links: PLAN.md (v4.1) issues: #1–#18

Problem

Umar wanted a "mission control": one place to watch (and act on) the significant activity across his dev stack and AI agents, instead of tab-hopping across Slack, Asana, GitHub, Drive, AWS, Vercel, and the Claude Code + Codex IDE panels. The challenge was not the idea, it was scoping an ambitious vision into something shippable without a rewrite, while honoring a strong supply-chain caution (he dropped a dependency over it, citing the axios + LiteLLM incidents).

Concrete pains driving it: costly/risky agent actions caught too late (a GPU left billing); buried Slack/Asana/GitHub items; losing the thread on what an agent did across sessions; and (biggest) context-switching so heavy you cannot tell what state anything is in. A later, sharper pain became the v1 wedge: you cannot see how much agent quota you have left, so you waste use-it-or-lose-it windows or hit limits mid-task.

Solution

We brainstormed the vision, then deliberately narrowed v1. Decisions, with the why:

  • Product = Mission Control; v1 beachhead = a cross-agent quota panel. Four live gauges (Claude 5h/weekly, Codex 5h/weekly) showing remaining % + reset countdown, framed to maximize usage. It is the highest-ROI, lowest-risk first slice and it builds the agent-data pipeline (local reads, event store, live UI, token security) that the watch-feed, integrations, and work-threads all reuse.
  • AWS tabled. The read-only AWS cost/idle radar was the most differentiated corner, but the owner removed it from v1 (no kill switch, no infra control). It returns later as a read-only watch.
  • Quota mechanism. Codex is exact on disk (rate_limits in the rollout JSONL). Claude is a read-only GET /api/oauth/usage using the OAuth token from the macOS Keychain (via /usr/bin/security); no quota-spending probe; degrade if unavailable. Never hardcode plan caps; read the vendor's number.
  • Stack = Next.js/TS (web/) + Python/FastAPI (api/). Heavier than four gauges need, but the full product is a real app, so build on the real stack now. Dependency principle relaxed from "near-zero" to "lean + audited + pinned" (mainstream frameworks OK; unmaintained forks not). We dropped a planned fork (claumon) and build the quota slice from scratch.
  • Agent config. Repo + CLAUDE.md rules; hooks (guard-secrets blocks secret read/write, format, and a verify-gate Stop hook); commands (/check, /dev, /spike, /fix-issue, /handoff); skills (quota-collector, teach).
  • Anthropic best practices applied (verify-with-evidence, JIT planning, context hygiene, fresh-context review, headless budgets, self-improvement) after verifying them against official docs and fact-checking X-attributed stats.
  • Token efficiency: use native prompt caching (+ Batch API); evaluated and skipped Headroom (in-band on our tokens, heavy deps, ~3% real bill savings on Claude traffic).
The plan went through six Codex review passes and lands at v4.1, verdict "ship as the v1 plan". Edge cases are caught in PLAN.md as caveats C1–C7 (gray-area Claude endpoint, Codex stale-after-reset, Keychain subprocess, CLI-format drift, etc.).

Context

Why it matters: this is the foundation every later surface stands on, and the principles (read-only/non-invasive, tokens-never-leak, lean+audited deps, vendor-neutral, degrade-never-lie) are now enforced by config, not just prose.

What is next: the half-day spike (issues #1–#2) — prove the Claude quota path via Keychain first (the only real unknown), then Codex. Then scaffold web/ + api/ (#3–#4) and build the v1 gauges.

Open decisions: menu-bar vs web-tab form factor; alert channel beyond local OS notification; v2 surface order after Slack; when to un-table AWS.

Roadmap: v1 quota → v1.5 watch-feed + consumed-cost metering + jump-to-agent → v2 collaboration surfaces (Slack first) + Vercel + Timeline → v3 work-threads → later AWS watch + shaping hooks.
