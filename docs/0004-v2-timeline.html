0004 — v2 unified Timeline

2026-06-23 · Shipped (web view) · Umar + Claude

area: web PLAN.md §1, §8 issues: #16 (slice a)

Problem

v1.5 (entry 0003) built the event-store spine and GET /api/timeline, but the "what just happened across everything, chronologically" view (PLAN.md §1) had no UI. v2 (#16) has three parts — (a) the Timeline, (b) a Slack collector, (c) a generalized action endpoint. Only (a) is buildable and verifiable with evidence inside the ephemeral cloud container: it is pure-local on the event store and needs no SaaS credentials. (b)/(c) need the owner's live Slack token, so they stay handed off. So this session built (a).

Solution

A read-only reading view at /timeline (web/app/timeline/page.tsx), plus a Dashboard / Watch / Timeline nav across all three pages. It polls /api/timeline?limit=500 every 15 s and renders the normalized rows newest-first. Same design tokens as the dashboard/watch (one product).

  • Vendor-neutral rows. A left severity rail (info/warn/crit dot), a surface chip + a kind chip (snapshot/status/throttle/headroom), a human summary, and relative time (absolute on hover). No surface is first-classed; pills are derived from the data present.
  • Tolerant summaries at the store boundary. payload is Record<string, unknown>, never trusted: small asNum/ asStr readers narrow each field. Codex snapshot → "5h 62% · weekly 81% left" (from remaining_pct); Claude → "N messages · ~$X est."; collector status → its degraded reason; alerts → their message. Unknown shapes fall back to the type label — degrade, never crash.
  • Gotcha baked into the summary: token counts are <redacted> in the store. scrub_payload's key matcher includes "token", so every *_tokens count is masked on persistence (safe over-redaction). The live dashboard (/api/quota, unscrubbed) still shows real totals; the Timeline can't, so the Claude summary leans on estimated_cost_usd + events, which survive. This is the honest signal, not a bug to "fix" by weakening redaction.
  • Key-collapse, not adjacency. The 30s heartbeats (claude snapshot + codex status) interleave by surface, so adjacent-identical collapse never fires. Instead, identical events (surface|type|severity|summary) fold into one row with a ×N count and a first-seen tooltip, ordered by newest occurrence (input is newest-first, so a key's first sighting is its newest). Any genuine change (e.g. rising cost) is a new key → its own row. The collapse key is also the React key — guaranteed unique, no dup-key warnings.
  • Degrade-never-lie states. API-down notice (copied from the shipped pattern), a degraded banner when the store read fails, and an empty-state card ("Open the Dashboard to start the live feed — events accrue while it is open") since rows only accrue while an SSE client (the dashboard) is connected.
Verified live with Playwright (light + dark) against this container's real Claude usage plus synthetic Codex windows/alerts: rows render with the right severity/chips/ summaries; the surface filter narrows correctly; ×N collapse compresses heartbeats (6 rows from ~14 raw); the empty state renders; zero console warnings. web lint + typecheck clean; api untouched (95 tests, 100%).

Context

Run it (unchanged): uv --directory api run uvicorn missioncontrol.main:app --host 127.0.0.1 --port 8787 + pnpm -C web dev, then open /timeline. No new env, no new deps.

Known gap: web/ still has no test harness (vitest/coverage not scaffolded — issues #3/#4/#14), so the Timeline's pure helpers (summarize, collapse) are typecheck-verified and evidence-verified but not unit-tested. The api side keeps its 100% gate. Standing up vitest is its own small PR.

Next (v2, still handed off): (b) the Slack collector — same pure-parser + fixtures pattern as transcript.py/codex.py, emitting slack.* event rows that this Timeline already renders generically. Needs the owner's Slack token for live wiring and a scope decision (mentions/DMs vs. channels — the "buried items" pain #3). (c) the generalized action endpoint — best done with Slack's first "act" action so the abstraction has >1 user (the /api/jump token+Origin pattern is the template). Then v3 (#17) work-threads (schema already carries work_thread_id) and AWS (#18).
