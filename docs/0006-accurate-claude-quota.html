0006 — Accurate Claude quota + honest stale windows

2026-06-23 · Shipped (Claude quota opt-in; verified via fixture) · Umar + Claude

area: api + web PLAN.md §6, §9 issues: #16

Problem

The owner compared Mission Control to Claude's own usage page and called it "wrong results": the dashboard showed Claude as consumed tokens (248M / ~$269), not the remaining quota the usage page shows (session 36% used, weekly 12%). Root cause (confirmed by grepping ~/.claude): the transcript carries only consumed counts (message.usage); the remaining windows live only behind the OAuth usage endpoint, which v1 deliberately deferred to stay credential-free. Two more accuracy notes followed: confirm whether Codex really has a 5h window, and show long resets in days.

Solution

  • Claude remaining-window quota (the fix). claudequota.py (pure, 100%): the OAuth usage JSON → Windows, remaining = (1 - utilization) * 100; tolerant of fraction-vs-percent, int/float/ISO resets_at, and odd shapes (degrade, never guess). The dashboard now shows Claude 5h + weekly gauges (primary, identical to Codex) and keeps the consumed/cost meter as a secondary detail.
  • The credential boundary, held. claudecreds.py is the ONE isolated reader (coverage-omit seam): it reads the Claude OAuth token (macOS Keychain via /usr/bin/security fixed-argv no-shell; Linux ~/.claude/.credentials.json), holds it in memory for a single read-only GET /api/oauth/usage, and returns numbers, never the token — errors swallowed/redacted, 60s cache. It is opt-in (MC_ENABLE_CLAUDE_QUOTA=1) so credential access is always explicit; off by default the dashboard says exactly which flag to set.
  • Codex 5h is real; the bug was staleness. Confirmed via OpenAI docs that Codex has a 5h rolling window + weekly. The "98% · resets now" the owner saw was a stale window: after reset the rollout still shows the pre-reset percent. Now codex.py (and claudequota.py) mark a window stale once resets_at has passed, alerts.py skips stale windows (no false throttle), and the gauge shows a muted "stale / awaiting reset" state instead of a misleading number.
  • Helpful countdowns. Resets over 24h render in days ("2d 16h", "1d 20h"). The Timeline's Claude summary now prefers the real remaining windows when present.
Verified: api 118 tests, 100% on pure logic, ruff + mypy clean; web lint + typecheck + build clean. The dashboard was rendered against a realistic fixture (Claude 64%/88% — matching the usage page's 36%/12% used — a stale Codex 5h, and day-scale resets) since the live credentialed fetch can't run here (agents must not read the token; the container proxy blocks the endpoint).

Context

To use it live (owner's machine): set MC_ENABLE_CLAUDE_QUOTA=1 in api/.env and restart. The reader auto-detects Keychain (macOS) or ~/.claude/.credentials.json (Linux). This is the deferred PLAN.md §6 path; it was the "only real risk" the spike flagged, so verify the four numbers look right and that no token ever appears in logs/UI.

Snapshot shape change: the claude object is now additive — it keeps all consumed fields and gains windows, quota_available, quota_degraded. Alerts/events treat Claude windows like any surface (vendor-neutral: Claude can now raise throttle/headroom alerts too).

Open / deferred: the generalized action endpoint + Slack mark-read (paused mid-build when this came in — slacksource.py read seam shipped in 0005; the write seam was removed and is a clean restart). The usage endpoint's exact field names are taken from PLAN.md §6 and parsed tolerantly; if a real response differs, the parser degrades rather than lies — adjust _WINDOWS / _fraction once seen against live data.
